Run your applications in Karaf Runtime

Karaf Runtime is a modern and polymorphic application runtime. It's a lightweight, powerful, and enterprise ready.
By polymorphic, it means that Karaf can host any kind of applications : WAR, Spring, OSGi, and much more.
Karaf can be use as a standalone immuatable runtime, or in a mutable runtime that you can manage remotely.

Key features

Hot deployment

You can directly drop your applications in the Karaf deploy folder, they will be directly deployed for you. You can also create your own deployer.

Since 4.x

Dynamic configuration

All configurations (for Karaf itself and applications) are located in the Karaf etc folder. All changes in the configuration files are taken on the flyv: no need to restart.

Since 4.x

Logging system

Karaf uses a centralized logging back end, supporting popular frameworks (log4j, slf4j, logback, ...).

Since 4.x

Provisioning

Karaf provides a very convenient way to provision applications: the Karaf Features.

Since 4.x

Shell Console

Karaf provides a complete Unix like shell console, allowing to manage your container and applications. This shell supports completion, contextual help, key bindings, and much more.

Since 4.x

Remote management

Karaf embeds a SSH server allowing you to remotely access the shell using any SSH client. On the other hand, Karaf also provides JMX MBean server allowing to manage the container using any JMX client.

Since 4.x

WebConsole

In addition of ssh and JMX, you can also manage Karaf Runtime using a simple browser thanks to the Karaf WebConsole.

Since 4.x

Security

Karaf fully supports JAAS based security framework. It also supports a complete RBAC system for shell commands and JMX objects. You can directly use this security layer in your own applications.

Since 4.x

Instances management

You can manage several children instances inside the Karaf Runtime root instance. It's a very convenient way to test applications or configurations without impacting your existing running instances.

Since 4.x

Docker & Cloud ready

Manage your Docker containers and images via Karaf shell console, provisionning a running instance in the simpliest way.

Since 4.x

10 good reasons to use Apache Karaf as your runtime

It’s extremely modular:
think Tomcat is to web applications but with *just* the stuff you need nothing more. But if your needs grow over time you can easily add whatever you need.
It can be small:
Karaf has a minimal distribution that basically is an equivalent of a minimal Linux distribution. You just start it and install what you need using an internet connection.
It’s an Apache project:
you know it is a truly independent and is backed by a foundation that has more than proven it can deliver high quality and reliable software infrastructure, while being completely transparent at the same time.
It promotes software development best practice:
by using OSGi at its core, developing applications in this runtime promotes loose coupling of software services that will make maintenance and evolution much easier. Always remember that shortcuts in software development are usually something you end up paying for in the long term. It is better to start with a good software architecture from the start and Karaf will provide you with exactly that.
Don’t let OSGi scare you:
you can use Karaf even if you don’t know how to build an OSGi bundle because you can deploy WAR files as you would on a Tomcat install or even using Spring as a container (although this is not the best practice as it does add some unnecessary overhead).
It can really scale:
some large IoT installations of Apache Karaf have over 17000 instances, and if you have clustering needs you can use Apache Karaf Cellar to have inter-node communication as well as deployment.
It works will all the technologies you know and love:
if can work with all the Java EE libraries or more modern one such as GraphQL. And it’s relatively easy to add a new one if it’s not available out of the box.
It supports bundle hot deployment:
basically you can setup watching on a bundle on a local Maven repository, which means that if you just rebuild your module in your source code it will automatically re-deploy your changes, without needing to restart anything. This makes development highly productive, in similar ways to what exists on NodeJS servers.
It has a powerful built-in SSH shell command line console:
this console is very useful for developers as well as DevOps working with Karaf. You can either remotely or locally connect to the shell using any SSH client and then use auto-completion, color highlighting, command history and even scripting to perform whatever operations you need. It is also very easy to implement your own shell commands with just a few annotations on a Java bean.
It has a proven track record:
Apache Karaf is used by many companies, ranging from small to large, to deliver solutions ranging from embedded IOT implementations to large clusters delivering mission critical services.

You have to migrate to Apache Karaf!

you’re an Apache Felix user

You’ll probably end up re-inventing Apache Karaf at some point because you’ll need:

  • Logging
  • Startup / shutdown scripts / OS native integration (Windows, Linux)
  • Docker integration
  • A more powerful and more secure (SSH) shell
  • Proper app provisioning (Apache Karaf features)
  • Clustering (Apache Karaf Cellar)
  • Java EE technologies integration (JPA, Web Applications, etc)
  • Proper injection support such as Blueprint, DS with annotations and CDI
  • JMX integration

...too much more to list them all

you’re a Spring Boot/Spring Framework user

  • You want a cleaner (in terms of modularity and separation of concerns) framework to write your applications with.
  • You want to make sure your modules are independent from one another.
  • You want to be able to put more functionality into a JVM, instead of adding more containers that each up more resources.
  • You want to be able to iterate fast by hot-deploying code instead of restarting the whole application for each change.
  • You want to use a framework coming from an independant foundation, that is not highly dependant on a company’s livelihood.
  • You want to have a lightweight application archive.
  • You don’t want to embedded bunch of jar dependencies.
  • You want to have more flexibility about dependencies upgrade.
  • You want to choose the JSR implementation framework at runtime and don’t make your code locked by one.

you’re a NodeJS user

  • You want to switch to a Java platform.
  • You want to keep some of the dynamic advantages of Javascript while working in Java.
  • You want to have an proven application runtime based on proven standards (OSGi).
  • You want to benefit from the type safety and protection that Java Virtual Machines offer.
  • You want to be able to use fine-grained authorization and authentication standards (JAAS).

Manage bunch of clustered instances with Karaf Cellar

You have bunch of Karaf Runtime instances running ? You want to manage those instances as one, spreading the configuration, deployment, etc ? Karaf Cellar is for you.
Karaf Cellar is a clustering solution for Karaf. It allows you to manage multiple instances, with synchronization between the instances.

Key features

Discovery

Each Karaf node is discovered automatically by the others, supporting different mechanisms (multicast, unicast, whiteboard, ...).

Since 4.x

Cluster groups

You can target the synchronization on a subset of nodes using cluster groups.

Since 4.x

Sync and distribute resources

Cellar is able to synchronize and distribute applications (features, bundles, non-OSGi application), configuration, or local event.

Since 4.x

DOSGi

Cellar support DOSGi (Distributed OSGi), allowing you to implement remote call between your applications.

Since 4.x

Extras

Key features

Collectors

The collectors harvest the monitored data (JMX metrics, log messages, ...).

Since 1.x

Dispatcher

A dispatcher (powered by OSGi EventAdmin) forwarding the collected data to the appenders and SLA.

Since 1.x

Appenders

The appenders receive the collected data and store data into a back end (elasticsearch, cassandra, JDBC, ...).

Since 1.x

SLA

SLA (Service Level Agreement) is a special kind of appender, checking the collected data, and eventually raising an alert (to a back end).

Since 1.x
